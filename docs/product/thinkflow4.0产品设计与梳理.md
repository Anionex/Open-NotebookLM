# Part A：产品形态与交互方式

## 1. 定位（总）

> **ThinkFlow 是一个"对话-理解-产出"一体的工作空间。** 用户和 AI 聊天产生的价值不再流失——它被凝结进文档、变成可查可复用的资产，最终衍生出 PPT、报告等交付物。

Slogan："对话即梳理，梳理即产出"。

核心理念一句话：**对话是流态的思考，文档是固态的理解，产出是对外的表达**。产品的工作就是把这三者无摩擦地打通。

## 2. 实体架构

四层实体，从外到内：

**工作空间**是任务容器——一个工作空间对应一件要做的事（如"宠物鲜粮竞品分析"、"Transformer 系统学习"）。工作空间不分模式，是通用的。

**素材**是外部输入（PDF、URL、Excel、图片、Word），**工作空间内全局共享**。素材有"激活"开关，决定是否参与当前对话的上下文。

**对话**是每次子任务的交互流。一个工作空间下可以有多个对话（"竞品 A 研究"、"定价模型学习"、"核心洞察梳理"），每个对话独立但共享同一个素材库和文档库。

**文档**是理解的凝结。一个工作空间下可以有多份文档，分四种类型：

| 类型     | 用户动机               | 结构              | 常见场景                         |
| -------- | ---------------------- | ----------------- | -------------------------------- |
| **探索** | 我在收集               | 条目流、无结构    | "帮我分析"、"研究一下"、“找出来“ |
| **学习** | 我要弄懂               | 有固定字段 schema | "我想学 X"                       |
| **梳理** | 我要整理（工作笔记本） | 自由章节          | "这些整理一下、我要完整文档"     |
| **产出** | 我要交付               | 目标格式骨架      | "做份 PPT / 思维导图/.."         |

**探索是默认入口**，工作空间进去时默认有一份 Inbox 探索文档。

## 3. 界面布局

三栏：

**左栏**是导航，三个 tab（对话列表 / 素材 / 产出列表）。

**中栏**是当前对话，消息流 + 输入区。

**右栏**是当前文档（可收起）。

顶部有工作空间切换和设置；布局跨模式统一，没有任何"模式切换"UI。

## 4. 对话与文档的两种关系⭐️

同一个对话框，可以同时和**多份文档发生两种不同的关系**：

**关联（右上角，多选）**——AI 回答时读取这些文档作为参考上下文，**不修改它们**。用户心智上这是"给 AI 提供参考资料"。点右上角的📎图标展开一个面板，里面勾选要参考的文档；激活的素材是工作空间级的默认关联，不需要重复勾选。

**编辑（对话框下方下拉，单选）**——指定这次对话要修改的目标文档。**下拉框默认是"无"**，选中某份文档后，对话进入"编辑模式"——用户的指令会被 AI 解读为"如何修改这份文档"。

这两个维度独立，所以组合有四种状态：

- **纯对话**（无关联、无编辑）→ 像普通聊天（但可以通过勾选回答来得到文档）
- **参考型**（有关联、无编辑）→ "结合我的学习文档回答这个问题"
- **改稿型**（无关联、有编辑）→ "把这段改简洁"
- **组合型**（有关联、有编辑）→ "参考素材 A 和梳理文档 B，把产出文档 C 的第 3 节补上"

**组合型是真正的生产力飞跃**——它把 Claude Code 那种 "read context + edit target" 的模式带进了知识工作场景。用户可以用一条指令，让 AI 跨文档工作，意图被交互结构显式表达，不需要在 prompt 里啰嗦解释"哪份是参考、哪份要改"。



## 5. 用户使用模式

### 5.1 三种典型起点（最顶层的总览）

用户进到一个工作空间，起点基本是三类：

**探索式起点**是最常见的（约 70%）。用户进来就开始聊，"帮我分析这份论文"、"研究一下这个市场"。他不一定知道最后要什么。界面上默认已经有一份 Inbox 探索文档承接这些发现，不需要用户预设任何东西。（包括取数和分析功能、也放在探索这里）

**学习式起点**（约 15%）。用户目标明确要学会某个东西。他的动作是：新建学习文档 → 填主题 → 开始对话。AI 基于主题初始化学习文档的字段 schema。

**产出式起点**（约 15%）。用户目标明确要交付。他的动作是：新建产出文档 → 选子类型 → 填基础配置 → 进入对话 + 产出流程。

三种起点进入的是**同一个界面**，只是首次打开时默认聚焦的文档不同。用户随时可以切换或并行——聊着聊着觉得有个知识点要系统学，新建学习文档；学到一半要对外汇报，新建产出文档。

### 5.2 每种文档最初需要什么

| 类型     | 创建时必填                                                   | 创建后的初始内容                                             |
| -------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **探索** | 无（可起名或用默认时间戳）                                   | 空 Inbox                                                     |
| **学习** | 主题名                                                       | AI 基于主题初始化五字段：当前主线 / 已理解 / 未完成问题 / 薄弱点 / 支线；"当前主线"里有一段建议描述供用户修改 |
| **梳理** | 名字                                                         | 空白文档；可选"基于已有文档生成章节骨架"                     |
| **产出** | 子类型（PPT/报告/导图/播客）+ 3 个轻量问题（受众、长度/时长、基于哪些文档） | 见 5.6 产出流程                                              |

四种文档底层都是 markdown，只是初始内容不同。类型可以后改（发现搞错了不怕）。

### 5.3 从对话到文档：两种方式

**方式 A——全对话转文档** 对话右上角"📋 沉淀为文档"按钮，弹出选择类型 + 起名，AI 基于整个对话生成目标类型结构的内容。适合"聊完一大轮一次性沉淀"，或聊到某个节点想把完整理解拎出来。

**方式 B——消息选中推送** 在对话里勾选一条或多条消息（用户的问题和 AI 的回答都可选），底部浮出推送条：选目标文档（已有或新建）+ 处理方式（追加 / AI 整理 / AI 融合）+ 补充指令（可选）。适合"边聊边挑精华"。

两种方式共享同一个推送 popover 组件，只是入口不同。技术上就是一次 push 接口调用。

### 5.4 对话与文档的两种关系（4部分已列举）

同一个对话可以和多份文档发生两种**独立**的关系：

**关联（📎 右上角、多选、只读）**——AI 回答时会读取这些文档作为参考上下文，但不修改它们。激活的素材自动作为默认关联。

**编辑（✏️ 输入框下方、单选、可写）**——设定本次对话要修改的目标文档。用户的指令被解读为"如何修改这份文档"。

两个维度独立，组合出四种对话状态，其中"**多份关联 + 一份编辑**"是产品最强的模式——下一节会用到。

### 5.5 文档之间：转化与组合（被低估的第二条生产路径）

文档不止来自对话，**文档也可以来自其他文档**：

**1 对 1 转化** 每份文档右上角"转化为..."菜单：探索 → 学习 / 梳理 / 产出、学习 → 梳理、梳理 → 产出、任何文档 → 素材（回流作为新一轮对话的输入）、下载导出。转化产生新文档，保留引用关系，源文档不消失。

**N 合 1 组合** 用 5.4 的"关联 + 编辑"组合，就能实现"几份文档合成一份"：关联多份文档作为参考 + 新建或编辑一份目标文档 → 一条指令让 AI 综合处理。

典型场景：

- 三份零散的探索文档 → 关联 → 新建梳理文档 → 指令："把这三份整合成完整的竞品分析"
- 一份学习文档 + 一份梳理文档 → 关联 → 新建产出文档 → 指令："基于这两份做一版对外报告"
- 多份素材 + 已有梳理 → 关联 → 编辑产出的某一节 → 指令："参考这些把这节缺的数据填上"

这条路径是 5.4 那套交互结构的真正价值释放——用户不用把文档内容复制粘贴到 prompt 里、不用给 AI 解释哪份是参考哪份是目标，交互本身已经表达了意图。

### 5.6 得到产出的流程（三阶段）

产出是**先有一份可迭代的产出文档、再进入最终生成**的两段式流程。产出文档是**项目档案**，最终生成的 pptx/docx 是**输出物**——档案长期存在、持续打磨；输出物是档案在某个时点的定格。

**阶段 1：创建产出文档（AI 生成有内容的初稿）**

用户点"新建产出"或在已有文档右上角点"产出为..."，进入轻量创建表单：

- 子类型（PPT / 报告 / 导图 / 播客）
- 来源选择（可选，单选或多选已有文档）
- 一句话目标（可选，如"给投资人看的 15 分钟路演"）

确认后 AI **立即生成一份有合理默认内容的产出文档**——不是空骨架，不放 `[待补充]` 占位，就是一份能直接用的首版初稿。用户看到的是现成的内容，觉得哪里不对再改。

以 **PPT 产出**为例，初始文档大致是这样：

```markdown
# 宠物鲜粮竞品分析 · Q4 产品团队分享

## 目标与受众
给产品团队做 15 分钟的竞品洞察分享，帮助团队
快速理解鲜粮赛道的格局和三家主要竞品的差异化。

## 风格指令
简洁商务风，深色背景。每页一个核心观点，
多用数据对比表格，少用长段文字。字体用无衬线。

## 内容大纲
- 第 1 页 · 封面 | 宠物鲜粮竞品分析 Q4 | 标题+副标题+日期
- 第 2 页 · 市场规模 | 2800 亿大盘，鲜粮赛道年增 45% | 折线图
- 第 3 页 · 三家竞品全景 | A 内容 / B 渠道 / C 私域 | 三列卡片
- 第 4 页 · 获客策略 | A 成本 80，B 成本 150 | 对比表
- 第 5 页 · 定价策略 | 价格带 / 毛利 / 逻辑 | 数据表
- 第 6 页 · 核心差异 | 获客 / 定价 / 用户三维度 | 矩阵图
- 第 7 页 · 趋势信号 | 竞品 A 高端化苗头 | 文字 + 箭头
- 第 8 页 · 启示 | 三条给我们的机会 | 要点列表

## 素材补充
(这里放对话中补充进来的素材、图片描述、引用段落等)

## 备注
```

不同子类型初始内容不同——报告类是摘要+章节正文结构，导图类是树状节点，播客类是对话脚本+时间分配。用户看到的永远不是"空白要自己补"，而是一份可以直接用的初稿，改总比从零写快。

**阶段 2：在产出文档里持续迭代**

这份产出文档就是一份普通 markdown 文档，用前面所有的能力打磨它：

- 直接编辑内容
- 把它设为**编辑目标**，在对话中指令 AI 修改（"第 3 页扩一下"、"风格改成杂志风"）
- **关联**其他文档作为参考，让 AI 往里填内容（"参考那份梳理，把第 5 页数据填上"）
- 对话新发现随时推送进来
- 上传图片作为素材让 AI 写描述、推送到"素材补充"
- 风格想法直接写进"风格指令"字段，不需要额外外挂配置

一份产出文档可以挂在那儿几天慢慢攒，这是和"一次性生成"最本质的区别。

**阶段 3：最终生成（大纲 + 风格 + 渲染）**

用户觉得"可以生成了"，点文档右上角**"生成最终产出"**，进入三步最终生成：

**3.1 AI 全面读取**——产出文档全文 + 它关联/引用的来源文档 + 激活素材，一起进入上下文。

**3.2 大纲 + 风格确认**——AI 基于产出文档的"内容大纲"和"风格指令"区域，生成：

- 最终大纲：每页/每节的标题、要点、视觉建议，在可视化编辑器里列出来；用户可拖拽排序、增删、改内容，或在旁边用对话指令 AI 调整
- 风格提案：色系、字体、版式倾向，用户确认或微调

这一步是质控关，一般 1-3 分钟。

**3.3 生成最终文件**——确认后 AI 渲染成真实的 pptx / docx / mp3 / 思维导图文件，保存到工作空间的产出列表。

### 5.7 对话级的模型切换

对话输入框旁边有**模型选择器**（小下拉），用户随时切换本次对话使用的模型。默认是"推荐"——后端按任务复杂度和上下文长度自动选合适的。可选项包括已接入的各家模型（Claude Sonnet / Opus、GPT-4、DeepSeek、等等）。

切换是**对话级别**的，不是全局设置——同一个工作空间里，深度分析对话可以用 Opus，简单整理对话可以用 Haiku，成本和质量用户自己权衡。每个对话记忆它上次用的模型，下次打开默认沿用。

而在首页中：用户在设置里填自己的 API 端点（URL + key），接入后在模型选择器里像官方模型一样使用。

### 5.8 版本与回溯

每份文档自动保留版本快照，触发点：

- 每次推送前、每次 AI 编辑前（系统自动）
- 用户手动编辑保存时（防抖 3 秒）
- 用户手动命名版本（如"对标阶段完成版"）

文档右上角"↺ 历史"按钮打开版本列表，任意一版可预览、可恢复。（待确认、结合维护成本）

# Part B：技术与 LLM 提示词设计

## 1. 整体原则

一条核心设计原则贯穿所有：**用交互结构携带意图，让 prompt 保持单一目的**。

用户在界面上做的每一个动作（选关联、选编辑目标、选推送模式、选文档类型）都在**显式表达意图**。后端拿到这些结构化信息后，**组装对应的 prompt**，不需要再在 prompt 里啰嗦"这是参考、那是目标"。这让每个 LLM 调用都是单一目的、边界清晰，容易调、容易测、容易换模型。

另一条是**最小必要上下文**：每次调用只送真正需要的东西进去，不送"可能有用"的。探索推送不需要看梳理文档，产出生成不需要看其他对话。

## 2. 后端的能力清单

整个产品的 LLM 层就五类调用，其他都是组合或微变体：

**对话调用** — 用户在对话框发消息时触发。上下文取决于当前对话的状态（关联/编辑/纯对话）。

**推送调用** — 用户推内容到某份文档时触发。一次变换：源内容 + 目标文档当前内容 + 处理方式 → 新的文档内容。

**创建调用** — 用户新建文档或沉淀对话为文档时触发。根据类型+来源生成一份有内容的初稿。

**转化调用** — 文档间 1 对 1 类型转换时触发。输入源文档，按目标类型 schema 重组。

**生成调用** — 产出文档的最终生成。分两次：一次生成大纲+风格方案，用户确认后一次渲染成最终文件。

另有一个轻量的**主动建议判断**，以规则为主、LLM 为辅，后面单讲。

## 3. 上下文组装结构

每次对话调用，后端装一个统一结构的 prompt：

```
[系统身份]           固定，定义 ThinkFlow AI 的身份与通用原则
[工作空间元信息]      名称、简短目标（如有）
[激活素材]           被激活的素材摘要或片段
[关联文档]           当前对话勾选的参考文档（多份）
[编辑目标]           如果设置了，目标文档的完整内容 + 类型倾向
[对话历史]           当前对话的消息序列（按 token 预算裁剪）
[当前用户消息]
[响应格式要求]       纯对话或 JSON（编辑模式）
```

按 token 预算裁剪的顺序是：**对话历史旧消息 → 素材摘要 → 关联文档正文摘要化**。编辑目标和当前用户消息永不砍。

## 4. 四种对话状态与对应 prompt

四种状态由交互显式决定，后端看到的是两个字段：`associatedDocIds`（数组）和 `editTargetDocId`（单值或空）。

### 状态 A：纯对话（无关联、无编辑）

最简单。基础系统提示 + 对话历史。

**系统提示示例**：

```
你是 ThinkFlow 的对话助手。用户正在一个叫
「{workspace_name}」的工作空间里和你交流。

回答要求：
- 直接、清晰、信息密度高
- 如果回答里有多个独立的知识点、结论或数据，
  在内容内用 Markdown 小节或列表分开，方便用户
  后续选择性推送到文档
- 不主动提"要不要推送"之类的操作建议
  （这类由系统另行处理）
```

### 状态 B：参考型（有关联、无编辑）

在状态 A 基础上注入关联文档。

**追加片段**：

```
用户为本次对话关联了以下文档作为参考资料。
你的回答应当建立在这些资料之上，必要时引用，
但不应改写或覆盖用户的原意。

【关联文档 1 · 类型：梳理 · 标题：竞品分析】
{doc_content}

【关联文档 2 · 类型：探索 · 标题：零散发现】
{doc_content}

【激活素材摘要】
- 行业报告.pdf：{summary}
- 竞品B产品手册.pdf：{summary}
```

**关键细节**：如果关联的文档里有**学习类型**，额外追加一段"用户在这份学习文档里标注了薄弱点 X、Y，回答时酌情照顾"。这让学习文档作为上下文时能主动迁移到新对话。

### 状态 C：改稿型（无关联、有编辑目标）

这是最特别的状态——AI 的输出是结构化的编辑指令，不是自由对话。

**系统提示追加**：

```
用户已将「{target_doc_title}」（类型：{type}
{subtype ? '· ' + subtype : ''}）设为本次对话
的编辑目标。用户接下来的消息将是修改指令、
讨论或澄清。

目标文档当前内容：
---
{target_doc_content}
---

{type_instruction}  <-- 按类型注入的倾向规则，
                        见第 5 节

请严格按下列 JSON 格式返回（不要额外文本）：

{
  "reply": "在对话里展示给用户的回复（自然语言，
            说明你理解了什么、做了什么修改、
            任何需要用户确认的地方）",
  "edit": {
    "risk": "low" | "medium" | "high",
    "action": "patch" | "rewrite" | "noop",
    "patches": [
      {
        "type": "insert" | "replace" | "delete",
        "anchor": "定位锚点（原文片段或章节标题）",
        "content": "新内容（如适用）"
      }
    ]
  }
}

风险分级规则：
- low：小改动，如添加一个要点、修正错别字、
       补一条资料 → 直接应用
- medium：重写一段、改结构但不伤主干 → diff 
          展示让用户逐段确认
- high：大量删除、重构整份文档 → 强制确认后再应用

如果用户消息只是讨论不需要改动，返回 action:
"noop"。
```

**后端处理**：解析 JSON → 按 risk 分级走三种 UI 路径 → 应用成功后更新文档 + 存一版历史快照。

### 状态 D：组合型（有关联 + 有编辑，最强）

状态 B 和 C 叠加。系统提示包含两段显式分隔：

```
本次对话涉及跨文档工作。

【参考资料 — 只读，不得修改】
[关联文档列表]
[激活素材]

【编辑目标 — 你需要修改】
[目标文档类型、子类型、当前内容、类型规范]

【协作要求】
- 从"参考资料"中提取信息来支持对"编辑目标"的修改
- 所有引用参考资料的内容，在 reply 里说明出处
  （"参考了《竞品分析》第 2 节的定价表"）
- 不要把参考资料的内容"平移"到目标文档，
  要按目标文档的类型规范重组
- 冲突时以用户的本次指令优先

返回格式同改稿型（reply + edit JSON）。
```

这个状态的产品体感就是——用户一句"参考这些把第 3 节填完"，AI 读 + 整合 + 改 + 溯源一气呵成，用户不用在 prompt 里解释哪份是什么。

## 5. 四种文档类型的倾向片段

这些是上面"type_instruction"的具体内容，按文档类型注入。每份都短、聚焦。

### 探索类型

```
这是一份【探索文档】，用于收集对话中产生的
零散发现。

对这份文档的修改应该：
- 最轻处理，保持原始性
- 按时间顺序追加，每条带时间戳和来源标注
- 不主动合并、不主动去重、不主动结构化
- 即使内容看起来重复，也保留——用户可能想
  看到思路的演进

只有用户明确指示"整理一下"、"去重"、"合并"
时，才进行结构化处理。
```

### 学习类型

```
这是一份【学习文档】，用户正在系统学习某个
主题。它有固定的五个字段：
- 当前主线
- 已理解
- 未完成问题
- 薄弱点
- 支线

对这份文档的修改应该：
- 判断内容归属到哪一个字段
- 不要新建字段，不要改字段名
- 小改动（加一条薄弱点、把一条从薄弱点挪到
  已理解）标记为 low 风险
- 重写"当前主线"这类核心字段，标记为 medium
- 在对话中识别到用户的混淆、没讲清楚的点时，
  可以主动把该点加入"薄弱点"，但要在 reply 
  里说明"我把 XX 加到了薄弱点，因为你的说法
  看起来还有模糊"
- 用户在对话中说清楚了某个之前的薄弱点时，
  把它从"薄弱点"挪到"已理解"

对话本身也可以做费曼式检验——用户讲到关键
概念时，可以追问"那你能不能用自己的话讲一下
X？"，但频率要克制，不要每轮都问。
```

### 梳理类型

```
这是一份【梳理文档】，用户在把零散的理解整
合成结构化文档。

对这份文档的修改应该：
- 判断内容归属到哪个已有章节
- 如果不属于任何已有章节，不要自作主张新开
  章节——在 reply 里问用户"这部分不属于任何
  现有章节，要新开一节叫 X 吗？"，action 
  设为 noop 等待确认
- 推送/融合时要去冗、整合，不要简单追加
- 保留来源标注（"来自对话 XX"、"来自素材 
  XX"），用引用块格式
- 发现内容与已有章节冲突时，在 reply 里明确
  指出，让用户决定采信哪个
```

### 产出类型

```
这是一份【产出文档】，子类型：{subtype}
（PPT / 报告 / 导图 / 播客）。
用户的目标：{goal}
受众与风格：按文档内"目标与受众""风格指令"
两节执行。

对这份文档的修改应该：
- 严格按子类型的结构规范（PPT 是每页要点列表、
  报告是章节正文、导图是树状节点、播客是对
  话脚本）
- 推送进来的内容，判断归到哪一页/哪一节
- 按子类型调整内容形态：
  · PPT：短句要点，每页不超过 3 行文字
  · 报告：完整段落，数据要有支撑
  · 导图：节点短语，控制在 1-2 层
  · 播客：对话口语，带时间戳
- 不要改用户在"风格指令"里写的任何话——那是
  用户自己的创作意图，你只执行，不改写
- 如果某页/节还没内容、但对话里讨论过相关，
  在 reply 里提议"第 X 页还空着，我看到
  对话里聊过 Y，要我填上吗？"
```

## 6. 推送调用（独立于对话）

推送是一次**独立的 LLM 调用**，和对话循环解耦。这样推送快、便宜、可预测。

```
任务：把一段内容推送到一份文档。

【源内容】
{pushed_content}

【目标文档】
类型：{type}
子类型：{subtype 或无}
当前内容：
---
{target_doc_content}
---

【处理方式】{append | organize | merge}
【类型规范】{type_instruction}
【用户补充指令】{prompt 或无}

处理方式说明：
- append：直接追加源内容到文档末尾，仅加时间
  戳和来源标注，不改写
- organize：把源内容按目标文档的类型规范结构化
  后追加（如梳理文档就整理为章节片段）
- merge：把源内容融合到已有内容的相关位置，
  去冗整合

请返回更新后的完整文档内容（markdown），不要
返回其他文本。
```

为什么返回完整内容而不是 patch？因为推送是一次确定性的批处理，用户会看到完整结果（不是 diff 审核流程），返回完整内容最稳、最不容易出 bug。编辑模式才需要 patch（用户要逐段审核）。

## 7. 创建与转化调用

### 新建文档

不同类型走不同的 prompt。以产出为例（其他类型更简单）：

```
任务：为用户创建一份初版产出文档。

【产出子类型】PPT
【用户目标】{goal}
【来源文档】（若有）
- {source_doc_1.type} · {source_doc_1.title}
  {source_doc_1.content}
- ...

要求：生成一份有合理默认内容的 markdown 产出
文档，包含以下部分：
1. # {title}
2. ## 目标与受众（一段话，基于用户目标扩展）
3. ## 风格指令（一段话，给一个合理的默认，
   用户之后会改）
4. ## 内容大纲（按 PPT 页数展开，每页一行：
   "第 N 页 · 主题 | 核心要点 | 视觉建议"。
   如果有来源文档，从中提取实际内容填充；
   如果没有，给出一个通用的骨架填充）
5. ## 素材补充（空节，留给后续追加）
6. ## 备注（空节）

**重要：不要放 [待补充] 占位符。**给具体内容，
不合适用户会改。每项要有可用的默认值，让用户
看到的是可以直接开始调整的初稿，不是要填的
空白。
```

### 文档转化

```
任务：把一份【{source_type}】类型的文档转化
为【{target_type}】类型。

【源文档】
{source_doc_content}

【目标类型 schema】
{target_type_schema}  <-- 如学习文档的五字段、
                           产出文档的页结构

要求：
- 保留源文档的核心信息
- 按目标类型的 schema 重组
- 在转化后的文档末尾加一行：
  "> 📎 转化自：{source_doc_title}"
- 返回完整的 markdown
```

## 8. 产出的最终生成（两次调用）

### 第一次：生成大纲 + 风格方案

```
任务：为产出文档生成最终大纲和风格方案，供
用户确认。

【产出文档全文】
{output_doc_content}

【关联来源文档】
{sources}

【激活素材】
{materials}

请综合以上所有上下文，生成：

1. 最终大纲（JSON 数组，每项包含 title、
   key_points、visual_hint）
2. 风格方案（JSON 对象：color_scheme、
   typography、layout_style、visual_emphasis）

返回 JSON：
{
  "outline": [...],
  "style": {...},
  "missing": ["标注的缺失项列表"]  <-- 如"第 5 页
            的数据支撑在素材中找不到"
}
```

`missing` 字段很重要——它让用户在点"确认生成"前就知道哪里可能不完整，决定是补内容还是硬生成。

### 第二次：渲染成最终文件

这一次严格说不是 LLM 调用，是后端的渲染服务（pptx 用 python-pptx，docx 用 python-docx 等等）。**但**，渲染前可能需要一次轻量 LLM 调用来把每页/节的要点展开为完整内容——因为用户确认的大纲可能只是要点，实际 PPT 页上要有完整文字。这次调用用最便宜的模型（Haiku、GPT-4o-mini 之类），按大纲每一项独立调用或批量调用。

## 9. 主动建议的触发

原则：**能用规则的不用 LLM、能离线的不在线算**。

**规则层**：后端定时或事件驱动地检查几个条件，零 LLM 成本：

- 探索文档条目数 > 阈值（可配，默认 20）且 24 小时内没给过建议
- 学习文档的"薄弱点"字段从非空变为空
- 梳理文档所有 `[待补充]` 标记都清空
- 产出文档的"内容大纲"里填充率 > 80%
- 某对话最近 N 轮 AI 回复都没有被推送

**LLM 层**：规则触发后，用一次轻量调用生成具体建议文案：

```
基于以下状态，生成一句自然、不打扰的用户建议：

触发规则：{rule_name}
相关文档：{doc_title}
相关上下文：{brief_context}

要求：
- 一句话，20 字以内为佳
- 友好、可操作
- 最后给出 1-2 个具体动作选项

示例：
"这几条都在讲定价策略,要开学习文档深入吗？"
选项：["✦ 开学习文档", "稍后再说"]
```

用户点接受的建议，走对应的创建/转化调用；点忽略，后端记一笔"该类建议 24 小时内不再触发"。

## 10. 模型路由

对话级模型切换在前端是一个选择器，后端要实现一个**模型路由层**：

- 每次调用前，根据 `conversation.model`（用户选的）或路由规则（"推荐"模式时自动选）决定用哪个模型
- 统一的 API 抽象层（OpenAI compatible 最省事），向下接各家 provider
- 失败降级：用户选了 Opus 但额度不够时，提示并退回到 Sonnet
- 用户自定义端点：保存为 user-level config，格式 `{name, base_url, api_key, model_id}`，出现在模型下拉里

不同调用类型也可以**默认用不同模型**以省成本：对话用中等模型、推送用便宜模型、生成大纲用强模型、渲染展开用最便宜模型。用户的对话级选择只影响对话调用本身，其他后台调用由系统决策。

## 11. 一张总图

整个 LLM 层的脉络就这些：

| 调用类型         | 触发       | 上下文          | 返回格式                     | 模型档位建议   |
| ---------------- | ---------- | --------------- | ---------------------------- | -------------- |
| 对话（4 种状态） | 用户发消息 | 完整组装        | 文本 或 `reply+edit` JSON    | 用户选 / 中-强 |
| 推送             | 用户推内容 | 源+目标+规则    | 完整新文档                   | 便宜-中等      |
| 创建初稿         | 新建/沉淀  | 类型+来源       | 完整新文档                   | 中-强          |
| 转化             | 类型转换   | 源+目标 schema  | 完整新文档                   | 中等           |
| 生成大纲         | 产出确认前 | 产出文档+来源   | `outline+style+missing` JSON | 强             |
| 渲染展开         | 大纲确认后 | 每页要点        | 每页完整文字                 | 便宜           |
| 建议文案         | 规则触发后 | 规则+少量上下文 | 短句+动作 JSON               | 最便宜         |

**全部只有七种调用**。新增文档类型只是加一份"类型倾向片段"和一个 schema；新增产出子类型只是加一个渲染器。架构扩展性非常干净。
