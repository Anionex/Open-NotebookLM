# 对话参考文档与模块焦点逻辑总结

更新时间：2026-04-27

本文用于对齐 ThinkFlow 当前工作区里三类状态的关系：

- 输入框下方勾选的参考文档
- 右侧文档里的模块选择 / 焦点选择
- 聊天与推送时的 prompt / target 组装

## 核心结论

当前实现里，输入框下方的文档勾选和右侧模块选择是两条不同链路：

- 输入框勾选文档：表示“本次对话参考哪些文档”，写入当前 conversation 的 `source_refs`。
- 右侧点击模块：表示“当前显示文档的焦点模块 / 下次推送目标”，写入该文档的 `focus_state`。
- 普通聊天 prompt 目前只完整展开“活跃文档”，不会按输入框勾选文档的模块焦点展开正文。
- 推送到文档时，`target: focus` 会读取右侧活跃文档的 `focus_state`，但当前后端只取第一个 section 作为落点。

因此，当前体验里的问题是：用户以为“勾选了某个文档 + 右侧选中了某个模块”会让聊天精确参考该模块，但实际 prompt 里两者没有模块级 match。

## 当前实现

## 输入框勾选参考文档

输入框下方的文档 checkbox 来自右侧文档列表。

前端状态：

- `boundDocIds`
- `conversationSourceRefs`

触发逻辑：

- 勾选文档时调用 `toggleBoundDoc(documentId)`。
- 文档被写入当前 conversation workspace state 的 `source_refs`。
- 类型按文档类型区分为：
  - `document`
  - `output_document`

当前语义：

```text
这个文档会作为当前对话的参考来源。
```

当前不足：

```text
只记录到了 document id/title/type，没有记录“参考这个文档里的哪个模块”。
```

## 右侧模块选择

右侧文档被按 Markdown 标题拆成模块。

点击模块后，前端会更新该文档的 `focus_state`：

```json
{
  "type": "sections",
  "section_ids": ["section-主要贡献与效果-1"],
  "stash_item_ids": [],
  "description": "焦点：§ 主要贡献与效果"
}
```

当前语义：

```text
这个文档当前选中的模块，是下次推送/并入时优先修改的焦点。
```

当前不足：

- 这个焦点属于文档本身，不属于某条聊天消息。
- 如果输入框勾选的是另一个文档，右侧焦点不会自动作用到那个勾选文档。
- 多选 section 虽然前端 description 能显示多个，但后端推送落点当前只取第一个 section。

## 普通聊天 Prompt 组装

普通聊天发送时，前端先调用：

```text
POST /api/v1/kb/conversations/{conversation_id}/chat-context
```

后端由 `ThinkFlowChatContextService.build_context()` 组装上下文。

当前 prompt 结构大致为：

```text
[系统提示词]
你是 ThinkFlow 工作区里的 AI。文档只能通过用户显式推送动作被修改；普通聊天只回答问题。

[活跃文档]
标题：梳理摘要 1
{活跃文档全文，最多 8000 字}

[活跃文档当前焦点]
焦点：§ 主要贡献与效果

[对话来源]
- 来源 1: [document] 梳理摘要 1
- 来源 2: [material] Expert_attention...

[对话历史消息]
user: ...
assistant: ...

[用户新消息]
{用户输入}
```

当前关键点：

- `active_document_id` 对应的文档会被展开全文。
- `active_document.focus_state.description` 只作为文字说明进入 prompt。
- `source_refs` 里的 document 目前只作为来源列表出现，不会展开正文。
- `source_refs` 里的 document 也不会按自身 `focus_state` 抽取模块内容。

## 推送到文档的 Prompt / Target 组装

当用户把聊天内容推送到文档时，前端会根据右侧文档焦点决定默认 target：

```text
如果 documentFocusState.type == sections -> 默认 target = focus
否则 -> 默认 target = document_end
```

推送请求大致为：

```json
{
  "mode": "organize",
  "title": "AI",
  "prompt": "",
  "text_items": ["本次要沉淀的对话内容"],
  "source_refs": [...],
  "target": {
    "type": "focus"
  },
  "transform": "ai_append"
}
```

后端收到 `target.type = focus` 后，会读取文档 meta 里的 `focus_state`：

```text
focus_state.type == sections
  -> 转成 target.type = section
  -> 使用 focus_state.section_ids[0]
```

当前不足：

- 多选模块没有真正作为多个落点生效。
- `focus` target 和“输入框勾选参考文档”没有直接关系。
- 推送的 `source_refs` 主要记录消息来源和素材来源，未完整表达“参考文档选中模块”。

## 目标模型

建议把三类概念明确拆开：

```text
输入框勾选文档 = 聊天参考范围
右侧选中模块 = 当前文档焦点范围 / 下次推送目标
两者通过 document_id 进行 match
```

## 聊天参考规则

普通聊天 prompt 应该按以下优先级组装参考文档：

```text
1. 遍历当前 conversation.source_refs。
2. 对 type=document/output_document 的 ref，读取对应文档。
3. 如果该文档有 focus_state.sections：
   - 解析文档 Markdown section。
   - 只抽取选中的 section 内容。
4. 如果该文档没有选中模块：
   - 使用文档全文截断内容。
5. 如果 ref 指向的文档不存在：
   - 只保留来源标题，不阻塞聊天。
```

目标 prompt 示例：

```text
[参考文档 1]
标题：梳理摘要 1
参考范围：选中模块
选中模块：主要贡献与效果
模块内容：
## 主要贡献与效果
...

[参考素材]
- Expert_attention____ACL_v2 (2).pdf

[活跃文档]
标题：梳理摘要 1
当前焦点：§ 主要贡献与效果

[用户新消息]
...
```

如果勾选文档没有选中模块：

```text
[参考文档 1]
标题：梳理摘要 1
参考范围：全文
文档内容：
{全文截断}
```

## 推送目标规则

推送到文档时，右侧活跃文档仍然是写入目标。

建议规则：

```text
1. target=document_end：追加到文档末尾。
2. target=section：写入指定 section。
3. target=focus：
   - 如果 focus_state 有一个 section，写入该 section。
   - 如果 focus_state 有多个 section，进入多模块融合逻辑。
   - 如果没有 focus，退回 document_end。
```

多模块融合建议：

```text
当 focus_state.section_ids.length > 1 时：

- 后端解析出所有选中模块。
- AI merge prompt 明确给出：
  - 当前选中的多个模块标题
  - 每个模块原文
  - 新增信息
  - 要求只修改这些模块，不改未选中模块
- 输出完整文档全文，或输出 section patch 后再由后端合并。
```

短期更稳的实现：

```text
先只支持“单模块作为推送落点”。
如果用户选中多个模块，推送弹窗里提示：
当前选中多个模块，本次将并入第一个模块：xxx
```

中期目标：

```text
支持多个 section 同时作为 merge 范围。
```

## UI 对齐

输入框下方文档 checkbox 的文案建议从：

```text
对话将参考此文档
```

改为更明确：

```text
聊天参考：已选文档
```

如果勾选文档存在选中模块，可以显示：

```text
聊天参考：梳理摘要 1 / 主要贡献与效果
```

如果没有选中模块：

```text
聊天参考：梳理摘要 1 / 全文
```

右侧模块选择栏保持表达“推送焦点”：

```text
焦点：§ 主要贡献与效果
```

或对产出文档：

```text
确认模块：§ 主要贡献与效果
```

## 推荐实现步骤

## 第一步：聊天 prompt 按参考文档 focus 展开

修改后端 `ThinkFlowChatContextService`：

- 新增“读取 source_refs 中 document 内容”的逻辑。
- 对每个 document ref：
  - 读取文档 detail。
  - 读取 `focus_state`。
  - 用与前后端一致的 Markdown section 解析规则抽取 section。
  - 将抽取内容写入 `[参考文档]` 块。
- 保留当前 `[活跃文档]` 块，但避免与参考文档重复过多。

## 第二步：前端显示参考范围

修改输入框下方 checkbox 展示：

- 勾选文档时显示文档名。
- 如果该文档有焦点模块，追加模块名。
- 如果无焦点模块，显示“全文”。

## 第三步：推送多模块策略

短期：

- 如果 `focus_state.section_ids.length > 1`，推送弹窗明确提示实际落点。
- 后端继续只取第一个 section，但不要让 UI 误导用户。

中期：

- 后端 `_resolve_push_target` 支持 `section_ids`。
- `_merge_with_ai` 支持多 section merge。
- trace 记录多个 section id。

## 需要避免的混乱

不要把输入框 checkbox 和右侧模块焦点混成一个状态。

正确关系应该是：

```text
checkbox 决定聊天参考哪些文档。
focus_state 决定每个文档当前参考哪个模块，以及活跃文档下次推送改哪里。
prompt 组装时按 document_id 把二者 match 起来。
```

也不要让普通聊天直接修改文档：

```text
普通聊天只能回答问题。
文档变更只能通过显式“推送”动作发生。
```

## 验收标准

- 勾选某个文档后，普通聊天 prompt 中包含该文档内容。
- 如果该文档有选中模块，prompt 中优先包含选中模块内容，而不是只包含标题。
- 如果该文档没有选中模块，prompt 中包含该文档全文截断内容。
- 右侧选中模块后，输入框参考文档区域能显示当前参考范围。
- 推送到文档时，默认 target 能清楚表达“全文 / 文档末尾 / 当前模块”。
- 多模块选择不会在 UI 上暗示“全部会被改”，除非后端已经真实支持多模块 merge。
