# PPT 大纲：对话与工作台双向协作交互 PRD

> 版本: v0.1 | 日期: 2026-04-23 | 状态: 讨论稿

---

## 1. 设计原则

**双向协作模型**：左侧对话负责"大改"（结构调整、风格指令、AI 协商），右侧工作台负责"小改"（标题微调、要点编辑）。两侧修改同步到同一份大纲，互不阻塞。

核心规则：
- 对话产生的修改走 **draft → 推送** 流程（AI 改动需用户确认）
- 右侧手动编辑 **即时生效**（用户自己的改动不需要确认）
- 冲突时 **智能合并，draft 优先**
- 对话修改触发右侧 **自动导航到受影响页面**

## 3. 阶段一：大纲编辑（outline_ready）

### 3.1 右侧手动编辑 → 左侧同步

**触发**：用户在右侧直接修改大纲字段（标题、布局描述、要点、素材引用）

**行为**：
1. 修改即时保存到后端（debounce 500ms，调用 `PUT /kb/outputs/{id}/outline`）
2. 左侧对话流插入一条轻量系统消息：

```
┌──────────────────────────────────────┐
│ 📝 手动修改                           │
│ 第3页标题: "问题背景" → "核心问题"      │
│                          12:03       │
└──────────────────────────────────────┘
```

3. 修改记录写入 `outline_chat_sessions` 的 messages（role: "system"），AI 下次对话时能感知
4. 如果当前有 pending draft，手动编辑修改的是 **正式大纲**（非 draft），推送时会合并

**数据流**：
```
右侧编辑 → updateOutlineSection() → debounce → saveOutline() → 后端持久化
                                                    ↓
                                        左侧插入 system message
                                                    ↓
                                        写入 chat session messages
```

### 3.2 左侧对话 → 右侧同步

**触发**：用户在左侧对话发送大纲修改请求

**行为**：
1. 用户发消息（如"把整体风格改成商务汇报"）
2. 后端生成 draft_outline + draft_global_directives
3. 左侧对话展示 **内联 draft 卡片**：

```
┌──────────────────────────────────────┐
│ 🤖 助手                              │
│ 已生成候选修改，主要变更：              │
│                                      │
│ 📋 候选改动对比                        │
│ ┌──────────────────────────────────┐ │
│ │ ✏️ 修改 第1页 "封面"              │ │
│ │   标题: "研究报告" → "商务汇报"    │ │
│ │ ✏️ 修改 第3页 "数据分析"          │ │
│ │   布局: "图文并排" → "全图表"      │ │
│ │   要点: +2 -1                    │ │
│ │ ➕ 新增 第8页 "行动计划"           │ │
│ └──────────────────────────────────┘ │
│                                      │
│ 🏷️ 全局指令变更                       │
│ ┌──────────────────────────────────┐ │
│ │ + 整体风格: 商务汇报               │ │
│ │ + 配色方案: 深蓝主色调             │ │
│ └──────────────────────────────────┘ │
│                                      │
│ [推送这版]  [继续修改]  [放弃]         │
└──────────────────────────────────────┘
```

4. **右侧自动导航**到 draft 中第一个被修改的页面
5. 右侧进入 **draft 预览模式**：显示 draft 内容，但仍可编辑正式大纲中 draft 未涉及的字段

### 3.3 Draft 期间的冲突处理

**场景**：对话生成了 draft（修改了第1、3、8页），用户在右侧手动编辑了第3页标题和第5页要点

**推送时的合并策略**：

```
字段级合并规则：
├── draft 修改的字段 → 用 draft 版本（draft 优先）
├── 仅右侧修改的字段 → 保留右侧版本
├── 两边都没改的字段 → 保持原样
└── 同一字段两边都改了 → draft 优先，左侧显示提示

示例：
第3页.标题: draft="数据洞察", 右侧="核心问题" → 用 "数据洞察"（draft 优先）
第3页.要点: draft=未修改, 右侧=已修改 → 保留右侧版本
第5页.要点: draft=未涉及, 右侧=已修改 → 保留右侧版本
```

**推送后**：
- 左侧清除 draft 卡片
- 右侧退出 draft 预览模式，显示合并后的正式大纲
- 如果有字段被 draft 覆盖了右侧编辑，左侧显示提示：

```
┌──────────────────────────────────────┐
│ ⚠️ 合并提示                           │
│ 第3页标题的手动修改被 AI 版本覆盖       │
│ 手动版: "核心问题" → AI版: "数据洞察"   │
└──────────────────────────────────────┘
```

### 3.4 全局指令管理

**添加/修改**：仅通过左侧对话（如"所有页面用深色背景"、"删除配色规则"）

**展示**：右侧大纲顶部显示只读标签列表

```
右侧工作台顶部：
┌──────────────────────────────────────┐
│ 🏷️ 全局规则                          │
│ [整体风格: 商务汇报] [配色: 深蓝主色调] │
│                                      │
│ 💡 通过左侧对话添加或修改全局规则       │
└──────────────────────────────────────┘
```

### 3.5 对话 → 右侧导航联动

| 对话事件 | 右侧行为 |
|---------|---------|
| AI 返回 draft，修改了第3页 | 右侧自动滚动到第3页，高亮该页卡片 |
| 用户点击 draft 卡片中的"第3页" | 右侧跳转到第3页 |
| 推送成功 | 右侧刷新大纲，停留在当前页 |
| 用户在右侧点击某页 | 左侧不变（单向导航） |

## 4. 阶段二：逐页生成审阅（pages_ready）

### 4.1 对话驱动单页修改

进入逐页审阅后，左侧对话切换为 **逐页修改模式**：

**对话 UI 变化**：
- 标题变为："🎨 逐页审阅 - 第 N 页"
- 输入框 placeholder："描述这页需要怎么调整..."
- 顶部显示当前选中页的缩略图

**交互流程**：
1. 用户在右侧选中某页（如第3页）
2. 左侧对话上下文自动切换到第3页
3. 用户发消息（如"文字太多，精简一下"）
4. 后端调用 `POST /kb/outputs/{id}/pages/2/regenerate`
5. 生成新版本，右侧显示新旧版本对比
6. 用户在右侧选择版本 / 确认

**对话消息格式**：
```
┌──────────────────────────────────────┐
│ 👤 用户                              │
│ 文字太多，精简一下                     │
│                                      │
│ 🤖 助手                              │
│ 已重新生成第3页，主要变化：             │
│ · 精简了3个要点为2个                   │
│ · 移除了底部注释文字                   │
│                                      │
│ [查看新版本]  [查看对比]               │
└──────────────────────────────────────┘
```

### 4.2 右侧工作台行为

| 操作 | 触发方 | 行为 |
|------|-------|------|
| 选中页面 | 右侧点击 | 左侧对话上下文切换到该页 |
| 重新生成 | 左侧对话 | 右侧显示新版本预览 |
| 选择版本 | 右侧按钮 | 左侧显示确认记录 |
| 确认页面 | 右侧按钮 | 左侧显示确认记录 |

## 5. 阶段回退

### 5.1 pages_ready → outline_ready

**触发**：用户点击"返回大纲编辑"按钮

**行为**：
1. 弹出确认对话框："返回大纲编辑后，已生成的页面将保留为历史版本。确认返回？"
2. 确认后：
   - `pipeline_stage` 回退到 `outline_ready`
   - 已生成的页面快照保存到 `page_versions` 作为历史
   - 右侧切换回大纲编辑视图
   - 左侧对话切换回大纲对话模式
   - 对话插入系统消息："已返回大纲编辑阶段，之前生成的页面已保留为历史版本"

### 5.2 generated → pages_ready

**触发**：用户点击"继续调整"按钮

**行为**：回退到逐页审阅，保留当前生成结果作为基线版本

## 7. 消息类型汇总

| 类型 | role | 来源 | 内容 |
|------|------|------|------|
| 用户对话 | user | 左侧输入 | 自然语言修改请求 |
| AI 回复 | assistant | 后端 LLM | 文字回复 + draft diff 卡片 |
| 手动编辑记录 | system | 右侧编辑触发 | "用户手动修改了第N页XXX" |
| 推送结果 | system | 推送操作 | "已推送候选修改到正式大纲" |
| 合并提示 | system | 推送合并 | "第N页XXX的手动修改被AI版本覆盖" |
| 阶段切换 | system | 阶段操作 | "已进入逐页生成" / "已返回大纲编辑" |
| 页面操作记录 | system | 逐页审阅 | "已确认第N页" / "已选择第N页版本X" |

## 8. API 变更需求

### 8.1 新增/修改端点

| 端点 | 变更 | 说明 |
|------|------|------|
| `PUT /kb/outputs/{id}/outline` | 修改 | 支持 `manual_edit_log` 参数，记录手动编辑到 chat session |
| `POST /kb/outputs/{id}/outline-chat/apply` | 修改 | 支持 `merge_strategy` 参数，返回合并冲突报告 |
| `POST /kb/outputs/{id}/revert-stage` | 新增 | 阶段回退，保留历史版本 |

### 8.2 数据模型变更

```typescript
// 新增：手动编辑记录
type ManualEditLog = {
  page_index: number;
  field: 'title' | 'layout_description' | 'key_points' | 'asset_ref';
  old_value: string;
  new_value: string;
  timestamp: string;
};

// 新增：合并冲突报告
type MergeConflictReport = {
  conflicts: {
    page_index: number;
    field: string;
    draft_value: string;
    manual_value: string;
    resolved_to: 'draft' | 'manual';
  }[];
  auto_merged_count: number;
};

// 修改：OutlineChatSession.messages 支持 role: 'system'
type OutlineChatMessage = {
  id: string;
  role: 'user' | 'assistant' | 'system';
  content: string;
  meta?: {
    type: 'manual_edit' | 'stage_change' | 'merge_result' | 'page_action';
    edit_log?: ManualEditLog;
    conflict_report?: MergeConflictReport;
  };
  created_at: string;
};
```

## 9. 设计决策记录

| # | 问题 | 决策 | 说明 |
|---|------|------|------|
| 1 | 手动编辑 debounce 时间 | **500ms 保存，blur 时汇总记录** | 保存用 500ms debounce 保证不丢数据；左侧编辑记录在用户 blur（离开输入框）时汇总为一条，避免对话流过碎 |
| 2 | Draft 预览模式视觉区分 | **左侧色条** | 橙色左边条 = AI 修改页，绿色左边条 = AI 新增页，无色条 = 未修改页。轻量且不遮挡内容 |
| 3 | 合并提示侵入性 | **Toast 通知** | 推送时如有字段被 draft 覆盖，右上角弹出 toast（3秒自动消失），不打断流程。因为已选"draft 优先"策略，用户无需决策，轻量通知即可 |
| 4 | 逐页审阅对话历史 | **共享对话流 + 页面过滤器** | 默认显示所有页面的对话时间线，可点击 `[筛选: 第N页]` 只看特定页面相关消息。兼顾完整性和聚焦 |
| 5 | 手动编辑记录粒度 | **只在保存时记录一条汇总** | 每次 debounce 保存时在左侧插入一条汇总消息（如"手动修改了大纲: 第3页(标题、要点)、第5页(标题)"），最简洁，AI 仍能感知变更范围 |
