# ThinkFlow PPT：大纲先行 + Chat 迭代设计

日期：2026-04-22

## 1. 背景

根据 [docs/product/thinkflow4.0产品设计与梳理.md](/mnt/paper2any/szl/szl/prj/Open-NotebookLM/docs/product/thinkflow4.0产品设计与梳理.md)，ThinkFlow 的核心不应再被理解为“文件先整理成文档，再做产出”，而应是：

`来源 -> 对话澄清 -> 理解固化 -> 产出`

对 PPT 而言，最关键的产品心智不是“先准备很多中间对象”，而是：

- 用户选定来源
- 系统先生成一版可读的 PPT 大纲
- 用户通过 chat 慢慢把大纲改顺
- 大纲确认后再进入逐页生成和确认

当前仓库里已经存在 PPT 的三阶段链路：

- 大纲生成
- 大纲确认后逐页生成
- 最终导出结果

但交互重心仍然偏“产出工作台 + 多个辅助面板”，还没有真正把“chat 修改大纲”做成 PPT 的主流程。

本设计只聚焦 PPT，不处理报告、导图、播客，也不进行完整 ThinkFlow 4.0 信息架构重构。

## 2. 目标

本次改造的目标是把 PPT 流程明确收敛成：

1. 基于来源生成 PPT 初版大纲
2. 以 chat 作为修改大纲的主要入口
3. 以大纲面板作为当前 PPT 的可视化状态
4. 大纲确认后才进入逐页生成

用户进入 PPT 后应该强烈感知到：

- 我当前正在改的是这份 PPT 的大纲
- chat 不是普通问答，而是在帮我改这份 PPT
- 大纲改满意之后，才进入逐页生成

## 3. 非目标

本次不做以下内容：

- 不重做整个工作空间的左中右三栏信息架构
- 不引入完整的“关联参考文档 / 编辑目标文档”通用模型
- 不把摘要、梳理文档、产出指导彻底删掉或迁移
- 不统一报告、导图、播客的生成逻辑
- 不重新设计后端 PPT 工作流本身，只在现有 `output + outline + pages` 骨架上重构交互主线

## 4. 核心产品决策

### 4.1 PPT 的主对象是“大纲”

创建一份 PPT 后，当前工作空间里最核心的对象不再是“文档”或“产出工作台”，而是这份 PPT 的页级大纲。

这意味着：

- 进入 PPT 后默认显示的是大纲状态
- chat 默认围绕大纲展开
- 大纲是本阶段唯一需要反复调整的核心内容

### 4.2 Chat 默认解释为“修改当前 PPT 大纲”

在 PPT 的大纲阶段，用户输入：

- “把第 2 页换成问题背景”
- “增加一页行业格局”
- “这套太学术了，改成给老板汇报的口径”
- “把结论提前”

系统都应该优先理解成“修改当前 PPT 大纲”，而不是普通知识问答。

如果用户确实提出纯问答型问题，第一版不单独区分模式，而是统一仍作为大纲改写上下文中的辅助理解处理。换句话说，第一版优先追求产品主链清晰，而不是兼容所有边角场景。

### 4.3 大纲阶段与逐页阶段硬切分

PPT 仍保留三阶段，但语义收敛如下：

- 阶段 1：AI 生成初版大纲
- 阶段 2：通过 chat 持续迭代大纲
- 阶段 3：确认大纲后逐页生成与确认

阶段 2 和阶段 3 之间必须硬切分：

- 还在大纲阶段时，不开放单页生图和单页重做
- 一旦确认大纲进入逐页阶段，当前大纲转只读
- 若用户要大改结构，应重新开始一轮 PPT，而不是在当前流程中混合大纲重构和单页重做

### 4.4 不再以“摘要 / 文档 / 产出指导”作为 PPT 主信息架构

这些对象在底层仍可存在，但在 PPT 第一版的主界面里，不再占主导叙事。

PPT 主流程里用户首先看到和操作的是：

- 当前 PPT 大纲
- 与大纲相关的 chat
- 大纲确认与逐页生成阶段切换

“摘要 / 梳理文档 / 产出指导”只作为创建大纲时的增强上下文来源，不作为 PPT 主流程的中心面板。

## 5. 用户流程

### 5.1 创建 PPT

用户在当前 notebook 内：

1. 选定来源
2. 点击 `PPT`
3. 系统展示一次来源锁定确认
4. 确认后生成一版页级大纲
5. 自动进入 PPT 大纲阶段

创建完成后，系统直接告诉用户：

- 这是一版初稿大纲
- 接下来请直接在 chat 中说你想怎么改

### 5.2 大纲迭代阶段

用户进入 PPT 后的默认体验应是：

- 中栏：大纲 chat
- 右栏：大纲可视化
- 当前页可选中查看，但不强制用户手动填表

用户可以通过两种方式修改大纲：

- 主要方式：chat 提要求，由 AI 修改整套大纲或当前选中页
- 次要方式：直接在右栏对单页标题、要点、说明做手动微调

chat 是主入口，右栏手动改是辅助入口。

### 5.3 确认大纲

当用户认为大纲满意后，点击：

- `确认大纲，进入逐页生成`

确认后：

- 大纲冻结
- 系统切换到逐页生成阶段
- 右栏不再强调可编辑，而是转成只读大纲 + 当前页生成结果

### 5.4 逐页生成与确认

在逐页阶段：

- 系统按确认后的大纲生成整套页面草稿
- 用户逐页查看
- 用户可以对单页提出修改要求并重做当前页
- 用户可以切换历史页版本
- 用户逐页确认
- 全部确认后导出 PDF / PPTX

## 6. 交互设计

### 6.0 总体视觉结构

本次采用已确认的 `A` 方案：

- 前期使用独立的 `PPT 大纲` tab 完成大纲生成与 chat 迭代
- 后期逐页生成、改单页、确认版本，进入现有的 `产出工作台`

这意味着 PPT 的前半段和后半段会有明确分工：

- `PPT 大纲` tab 负责改结构、改页序、改每页定位
- `产出工作台` 负责逐页生成、单页重做、版本切换和最终导出

采用该方案的原因是：

- 更符合“先出大纲，再通过 chat 慢慢改需求”的产品叙事
- 能保留现有逐页工作台的价值，不必重做后半段
- 用户不容易把“改结构”和“改单页”混在一起

### 6.1 中栏：从“普通对话”改成“大纲对话”

PPT 大纲阶段的中栏 header 需要明确传达当前上下文：

- 当前对象：`PPT 大纲`
- 当前阶段：`大纲迭代中`

输入框提示词应改成类似：

- “直接告诉我怎么修改这份 PPT 大纲，例如：增加一页问题背景、把结论提前、改成适合投资人汇报的结构”

这样用户会自然把 chat 当成改大纲工具，而不是普通问答框。

### 6.2 右栏：默认展示 PPT 大纲，而不是摘要/文档/指导切换

在 `PPT 大纲` tab 内，右栏默认内容应是：

- 页级大纲列表
- 当前页详情
- 当前页的角色/布局说明
- 当前页要点
- 大纲确认按钮

右栏的目标不是当“另一个文档编辑器”，而是当“当前 PPT 结构状态面板”。

### 6.2.1 `PPT 大纲` tab 的信息布局

`PPT 大纲` tab 的布局应是：

- 中间主区域：大纲 chat
- 右侧：大纲状态面板
- 顶部：当前阶段说明与操作按钮

顶部需要明确告诉用户：

- 当前正在编辑的是这份 PPT 的结构
- 你可以继续聊天修改整套大纲或当前焦点页
- 满意后点击“确认大纲，进入逐页工作台”

因此，这个 tab 不属于“逐页产出工作台”的子状态，而是 PPT 的前置阶段界面。

### 6.3 选中某一页时，chat 默认作用于该页

当用户在右栏选中了某一页时，系统将该页视为“当前焦点页”。

此时 chat 的隐含目标优先是：

- 修改当前页

如果没有选中页，则默认修改整套大纲。

这条规则不需要做成复杂模式切换，只要在界面文案中明确告知即可。

### 6.4 移除独立的“AI 修改大纲 textarea”

当前已有单独的 `pptOutlineFeedback` 文本框用于 AI 修改大纲。它和 chat 重叠，应移除，避免出现两个“都能改大纲”的入口。

改造后：

- 所有 AI 大纲修改都从 chat 输入框进入
- 右栏只保留可视化编辑和阶段操作

### 6.5 大纲冻结后显示为只读快照

一旦进入逐页阶段：

- 大纲变成只读快照
- 用户仍可展开查看整套大纲
- 但不再允许在当前流程里修改大纲结构

界面要明确告诉用户：

- 当前阶段只改单页，不改大纲
- 若要重做结构，请重新开始一轮 PPT

### 6.6 确认大纲后的返回策略

该点已确认采用严格策略：

- 一旦用户点击“确认大纲，进入逐页工作台”
- 当前这轮 PPT 不再提供返回大纲继续改结构的入口

用户在逐页阶段仍然可以：

- 查看已确认大纲
- 基于已确认大纲改单页
- 切换当前页历史版本

但不能：

- 回到当前轮次的大纲编辑态继续改整套结构

如果用户需要重做结构，产品应引导其：

- 新建一轮新的 PPT
- 重新生成并修改一份新的大纲

## 7. 状态模型

### 7.1 输出对象仍沿用现有 `output`

本次不引入新的 PPT 主实体，继续沿用现有 `ThinkFlowOutput`。

保留字段：

- `target_type = ppt`
- `outline`
- `pipeline_stage`
- `page_reviews`
- `page_versions`
- `result`

### 7.2 新增“PPT 大纲对话历史”

为了支持 chat 驱动大纲修改，需要为每个 PPT output 增加一份独立的 outline-chat 历史。

第一版建议将其挂在 output 维度，而不是并入 notebook 的普通 conversation：

- 一个 output 对应一份 `outline_chat_history`
- 只记录围绕该 PPT 大纲的用户指令和 assistant 回复

这样可以避免把 notebook 普通问答和 PPT 大纲编辑混成一条历史流。

### 7.3 焦点页状态

前端增加一个明确语义：

- `activeOutlineSlideIndex`

这表示当前用户正在查看或修改哪一页。

chat 提交时：

- 若有焦点页，则优先按“改单页”理解
- 若无焦点页，则按“改整套大纲”理解

## 8. API 设计

### 8.1 保留现有接口

继续保留当前 PPT 相关接口：

- `POST /api/v1/kb/outputs/outline`
- `PUT /api/v1/kb/outputs/{output_id}/outline`
- `POST /api/v1/kb/outputs/{output_id}/outline-refine`
- `POST /api/v1/kb/outputs/{output_id}/generate`
- `POST /api/v1/kb/outputs/{output_id}/pages/{page_index}/regenerate`
- `POST /api/v1/kb/outputs/{output_id}/pages/{page_index}/confirm`

### 8.2 新增统一的大纲 chat 接口

新增：

- `POST /api/v1/kb/outputs/{output_id}/outline-chat`

请求字段建议：

- `notebook_id`
- `notebook_title`
- `user_id`
- `email`
- `message`
- `active_slide_index` 可选

服务端行为：

1. 读取当前 output 的 outline
2. 读取该 output 的 outline-chat 历史
3. 结合来源锁定快照、相关上下文，判断本次修改意图
4. 产出更新后的 outline
5. 生成一段 assistant 回复
6. 落盘新的 outline 与 outline-chat 历史

返回字段建议：

- `output`
- `assistant_message`
- `applied_scope`
  - `global`
  - `slide`
- `applied_slide_index` 可选
- `change_summary`

### 8.3 `outline-refine` 的处理策略

第一版不立即删除 `outline-refine` 后端能力，但前端不再直接使用它。

实现上有两种可接受方式：

- `outline-chat` 内部直接复用 `outline-refine` 的核心逻辑
- 或抽出一个共享 service，供二者共用

但前端主流程只保留一个 AI 改大纲入口，即 chat。

## 9. 前端改造范围

### 9.1 `ThinkFlowWorkspace.tsx`

重点改动：

- PPT 创建后进入独立的 `PPT 大纲` tab
- 移除独立的 PPT 大纲 AI textarea 交互
- 在 PPT 阶段让 chat 输入走 `outline-chat`
- 维护 output 级别的 outline-chat 消息状态
- 在大纲阶段渲染独立 tab 布局，在确认后切换进入现有逐页工作台

### 9.2 `ThinkFlowCenterPanel.tsx`

需要让中心 chat 面板在 PPT 大纲阶段显示专属语义：

- 输入框文案变化
- header 文案变化
- 允许显示“当前修改对象：整套大纲 / 第 N 页”

### 9.3 右栏面板结构

第一版不做全局右栏重构，但对 PPT 场景做局部重心调整：

- 当存在 active PPT output 且处于大纲阶段时，在 `PPT 大纲` tab 内优先渲染 PPT 大纲
- 弱化摘要/文档/指导切换对 PPT 主链的干扰

### 9.4 逐页阶段

保留当前逐页生成与确认能力，但界面文案要统一改为：

- 当前大纲已确认，只读
- 这里只改单页，不改结构
- 当前轮次不再返回大纲编辑

## 10. 提示词原则

### 10.1 大纲初稿生成

系统提示要强调：

- 不是直接生成最终 PPT 页面，而是生成“可供继续协作修改”的页级大纲
- 每页都需要有明确角色，而不是堆信息
- 优先给出合理结构，不留大量占位符

### 10.2 大纲 chat 修改

系统提示要强调：

- 当前任务是修改现有 PPT 大纲
- 不是重新自由回答问题
- 若用户聚焦某一页，优先修改该页
- 修改结果必须回写为结构化 outline
- assistant 回复应简洁说明本次改了什么

### 10.3 逐页阶段

系统提示要强调：

- 当前不能改整套结构
- 本轮只重做当前页
- 当前页要忠于已确认大纲

至少补以下回归：

- 中栏和右栏都明确处于 PPT 大纲阶段

验证：

- 在 PPT 大纲阶段发送 chat 消息后，会请求 `outline-chat`

### 11.3 焦点页改单页

验证：

- 不再显示当前轮次的大纲 chat 修改入口

- 不回退到大纲阶段

## 12. 风险与取舍

### 12.1 与现有 notebook chat 历史分离

这是刻意设计。

风险是用户可能觉得“为什么 PPT 里的 chat 和普通对话不是同一套历史”。

取舍理由是：

- 普通知识问答和 PPT 大纲编辑是两种不同任务
- 混在一起会让产品心智变脏

第一版优先保持任务边界清晰。

### 12.2 仍保留旧结构痕迹

由于本次不做完整 4.0 重构，底层仍然保留：

- output/document/guidance 等旧对象关系

这是接受的。

本次目标不是彻底重构信息架构，而是把 PPT 主流程先拉正。

### 12.3 用户可能想在逐页阶段继续改结构

第一版明确不支持。

若用户需要大改结构，应：

- 重新开始一轮新的 PPT

这样可以避免当前流程中“大纲改写”和“单页修图”相互污染。

## 13. 交付结果

本次设计完成后的产品体验应满足：

- 用户创建 PPT 后，第一反应是“我正在和 AI 一起改这份 PPT 大纲”
- chat 成为改大纲的主入口
- 大纲是当前 PPT 的核心对象
- 大纲确认后再进入逐页生成
- 单页重做与结构修改边界清晰

这将为后续扩展到报告、导图，以及更完整的 ThinkFlow 4.0“关联 / 编辑目标”模型打下统一基础。
